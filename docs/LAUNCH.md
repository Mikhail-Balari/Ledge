## Hacker News (body)

I built Ledge because I couldn't stop thinking about this: AI generates code all day, in millions of sessions, in languages designed for humans typing on keyboards in 1991.

Python was designed so a person could write it with their fingers. Its syntax carries 30 years of historical decisions — `**` for power, `//` for floor division, `@` for decorators, four ways to format a string. AI learned to use these because it had no choice. It inherited the languages humans made.

**Ledge flips the order of priority.**

Primary: designed to be generated by AI.
Secondary: designed to be read by humans.

The constraints that follow from this are surprisingly strong:

**One canonical form per concept.** Not two, not three. One. Python has 4 ways to format a string. JavaScript has 3 ways to declare a function. Ledge has exactly one way to do each thing.

**Every fallible operation requires an explicit fallback.** Not an option — required by the language. `divide(a, b) or 0`. `items[99] or "not found"`. `data["key"] or default`. You cannot write a Ledge program that crashes silently.

**AI is a language instruction, not an import.** `analyze("text") using sentiment`. No import needed. No API key in the code. The runtime handles the model. The program is model-agnostic.

**Reads aloud as English.** `for each item in items:` `define name as "Ledge"` `set count to count + 1`. Every statement sounds like a sentence.

## Reddit — r/ProgrammingLanguages

**Title:** Ledge — a language where the primary author is AI, not a human

**Body:**

I've been working on a programming language called Ledge. The premise: AI systems generate code all day in languages designed for humans. What would a language look like if we designed it for the inverse order of priority?

Core design decisions:

**No symbols with arbitrary meaning.** `->`, `::`, `{}`, `**`, `//`, `@` — these exist because someone chose an available key. They have no intrinsic logic. Ledge uses only mathematically universal symbols (`+`, `-`, `*`, `/`, `=`, `<`, `>`). Everything else is words.

**Exactly one form per concept.** Every alternative syntax is an opportunity for ambiguity. Ledge has one way to do each thing. No `def` vs `lambda` vs arrow functions. No `x++` vs `x += 1`.

**Error handling is syntax, not convention.** Every operation that can fail requires an explicit fallback as part of the expression: `divide(a, b) or 0`. You can't write a Ledge program that crashes silently without knowing it — the language structure makes this impossible.

**AI is a first-class primitive.** `analyze("text") using sentiment`. Not a library import. A language instruction, like `print` in Python.

Some code:

```ledge
# Variables
define name as "Ledge"
define version: number as 1

# Safe operations — or fallback required
define result as divide(total, count) or 0
define found  as items[99] or "not found"

# Functions
define fibonacci(n):
    if n <= 1:
        return n
    return fibonacci(n - 1) + fibonacci(n - 2)

# Higher order
define evens as filter(range(20), given n: modulo(n, 2) = 0)
show sum(evens)

# AI native
define sentiment as analyze("The product exceeded expectations") using sentiment
show sentiment   # → {tone: "positive", confidence: 0.94}
```

I'd love feedback on the grammar, the error handling model, and whether you think there's value in this separation between "AI-generation-first" and "human-typing-first" languages.

## LinkedIn post

**Title:** I built a programming language for AI to write

For the past several weeks I've been working on a question: if AI generates most of the code being written today, why is it still using languages designed for human fingers in 1991?

Today I'm releasing Ledge v0.1 — an experimental DSL designed primarily for AI generation, with human readability as a strong secondary constraint.

**The core design decisions:**

Every construct has exactly one canonical form — no alternatives, no ambiguity. AI systems benefit enormously from constraint: they generate better code when there's only one correct way to express something.

Error handling is mandatory by syntax, not convention. Every operation that can return "no result" — division by zero, missing array index, failed type conversion — has a built-in fallback operator. You cannot write a Ledge program that fails silently.

AI inference is a language instruction. `analyze("text") using sentiment` is as fundamental as `print("hello")`. No imports, no API keys in code, no library lock-in.

**What's in v0.1:**

Complete interpreter, bytecode compiler with VM, automatic formatter, LSP server for any editor, VS Code extension, browser playground, 150 tests passing, pip-installable.

This is a reference implementation. The goal is an LLVM-compiled native runtime under 100kb that runs on any processor — from cloud servers to embedded systems to edge devices.

If you're building with AI, curious about language design, or work at an organization thinking about infrastructure for AI-generated code, I'd love to know what you think.
